# AskFate 商业化改造 · 部署与运维说明

> 本文档对应「AskFate 功能与商业化改造需求」的交付物：服务端计费体系、按次消费、充值、安全扣次与流水、以及部署/环境变量/数据集合说明。

## 1. 改造概述

保留首页粉蓝宇宙视觉与「开始我的探索」入口，将站点从展示页升级为**可实际使用、可充值、可按次消费**的 AI 命理产品。

- 余额、扣次、流水、充值、退款**全部在服务端完成**，前端只读。
- 用户身份以「已签名 entitlement 的 deviceId」作为 userId；新用户首登自动赠送 `FREE_TRIAL_QUOTA=3` 次（仅一次，幂等）。
- 各功能消耗次数可在 `config/billing.ts` 配置：`reading1 / huangli0 / horoscope1 / tarot1 / liuyao2 / bazi3 / qimen3 / synastry3`。
- 充值套餐可在 `config/billing.ts` 配置：`lite ¥9.9=20 / standard ¥29.9=70 / deep ¥68=180`（后台 `recharge_packages` 集合优先）。

## 2. 环境变量

服务端通过环境变量注入凭证（**密钥不写死在代码里**）。本地开发可在 `.env.local` 填写；生产在 CloudBase 云托管「环境变量」配置。

| 变量 | 说明 | 必填 |
| --- | --- | --- |
| `TCB_ENV` | 云环境 ID（默认 `stain-d4geos14g9529a413`） | 是 |
| `TCB_SECRET_ID` | 腾讯云 SecretId（直连云数据库） | 是 |
| `TCB_SECRET_KEY` | 腾讯云 SecretKey | 是 |
| `AI_PROVIDER` | AI 供应商，如 `deepseek` | 是 |
| `DEEPSEEK_API_KEY` | DeepSeek API Key | 是（用 deepseek 时） |
| `ENTITLEMENT_SECRET` | entitlement token 签名密钥 | 是 |
| `WECHAT_MCH_ID` | 微信支付商户号 | 否（缺则走激活码临时方案） |
| `WECHAT_SERIAL_NO` | 商户 API 证书序列号 | 否 |
| `WECHAT_PRIVATE_KEY` | 商户 API 私钥（PEM） | 否 |
| `WECHAT_API_V3_KEY` | 微信支付 APIv3 密钥 | 否 |
| `WECHAT_NOTIFY_URL` | 支付回调公网地址 | 否 |
| `WECHAT_APP_ID` | 公众号/小程序 AppId | 否 |
| `WECHAT_PLATFORM_CERT` | 微信平台证书（PEM，验签回调用） | 否 |
| `FREE_TRIAL_QUOTA` | 新用户赠送次数，默认 3 | 否 |

> 微信支付 6 项参数**齐全**才启用真支付（`WECHAT_PAY.enabled`），否则创建订单返回 `pay_method:'manual'`，**绝不伪造支付成功**，保留激活码临时方案。

## 3. 数据库集合（迁移 / Schema）

使用 CloudBase 云数据库（文档型）。集合在首次请求时由 `bootstrapBilling()` **自动建集合并种子套餐**，无需手工建表；以下为集合与关键字段。

### `user_balances`（用户余额）
- `user_id` string（= entitlement.deviceId，唯一）
- `balance` number（剩余次数）
- `free_granted` boolean（是否已发首登赠送）
- `created_at` / `updated_at` number

### `usage_ledger`（流水，全部可追溯）
- `_id` string（= `${user_id}::${request_id}`，**幂等主键**）
- `user_id` / `feature` / `delta`（正数入账、负数扣次、退款为正）
- `balance_before` / `balance_after`
- `request_id` / `description` / `created_at`
- `refunded` boolean / `is_refund` boolean

### `recharge_orders`（充值订单）
- `order_id` / `user_id` / `package_id` / `amount` / `credits`
- `status`：`pending|paid|failed|closed|refunded`
- `payment_provider` / `provider_transaction_id` / `paid_at`

### `recharge_packages`（套餐，后台可改）
- `package_id` / `name` / `price` / `credits` / `bonus_credits` / `enabled` / `sort_order`

### `interpret_history`（解读历史）
- `user_id` / `feature` / `feature_name` / `summary`
- `result`（结构化计算）/ `ai_text`（AI 解读文本）
- `cost` / `request_id` / `created_at`

## 4. 部署步骤

```bash
# 1) 安装依赖（含云数据库 SDK 与 ws）
npm install

# 2) 本地构建（本机 NODE_OPTIONS 可能干扰 Turbopack，先取消）
env -u NODE_OPTIONS npm run build

# 3) 注入数据库凭证到云托管环境变量（仅首次或轮换密钥时）
python tcbr_ctl.py setenv          # 合并 TCB_ENV/TCB_SECRET_ID/TCB_SECRET_KEY 到 EnvParams

# 4) 部署到 CloudBase 云托管
node_modules/.bin/tcb cloudrun deploy -s stain --port 3000 --force

# 5) 确认最小实例数（避免缩容到 0 冷启动 503）
python tcbr_ctl.py setmin 1
```

> 部署后若 EnvParams 被重置，重复步骤 3 再 `detail` 复核即可。

### 4.1 部署关键修复：cloudbase SDK 与 Next 运行时不兼容（已解决）

初版部署后新建接口（`/api/balance` 等）返回 **500（空 body / ReadableStream 错误）**。排查结论分两层：

1. **依赖未进独立包**：`output: 'standalone'` 的 nft 追踪会漏掉使用动态 `require` 的 `@cloudbase/node-sdk`/`ws`——即使已在 `next.config.ts` 的 `serverExternalPackages` 声明，Next16 也不会把它们复制进 `.next/standalone/node_modules`，运行时 `require` 失败（部署前复查实测 `.next/standalone/node_modules` 内确实无 `@cloudbase`）。
2. **全局流冲突（根因）**：即便依赖就位，SDK 在 **Next 进程内**运行时仍抛 `Expected ... to be an instance of ReadableStream` —— Next 劫持了全局 `ReadableStream`/`fetch`，与 SDK 内部使用的 Node 原生流冲突。**已用纯 Node 脚本验证：SDK 在独立 Node 进程里完全正常，仅在 Next 运行时报错。**

**最终修复（架构级，已本地端到端验证）：把计费/数据库层拆成独立纯 Node 进程（sidecar）**
- `billing-service/index.mjs` + `billing-service/billing.impl.mjs`：纯 Node + `@cloudbase/node-sdk`，HTTP 暴露 `POST /billing`（RPC：`{op, args}`）。
- `billing-service/package.json`：**仅依赖 `@cloudbase/node-sdk`**；`Dockerfile` runner 阶段只为该目录 `npm install --omit=dev`，sidecar 自带 `node_modules`，`require` 向上解析到 `billing-service/node_modules`——**不污染 Next 的精简 standalone 包**，且容器里 guaranteed 有 SDK。
- `lib/billing.ts` 改写为**薄 RPC 客户端**，导出签名与实现版完全一致，所有路由/守卫无需改动。
- `Dockerfile`：`CMD ["sh","start.sh"]` 同容器拉起 sidecar（127.0.0.1:3100）+ Next（3000）；`start.sh` 先等 sidecar `health` 就绪再起 Next。Next 路由经 `http://127.0.0.1:3100/billing` 调用，SDK 永不在 Next 运行时内执行。

> 本地验证结果：经 Next 打 `/api/balance`，路由正常执行（无效 token 返回 402，有效 token 经 sidecar 调用 cloudbase），Next 日志 **0 条 ReadableStream 错误**，证明冲突已根除。

### 4.2 上线前的两个账户级阻塞（用户已于 2026-07-18 充值处理）

初版联调时发现 `stain` CloudBase 环境存在两项**账户/额度**问题：

1. **数据库读配额超限** `LimitExceeded.OutOfReadRequestQuota`：反复联调把免费读额度耗光；sidecar 会如实上报该错误（表现为接口 500「查询失败」）。已优化：`bootstrapBilling()` 改为进程内只执行一次，显著降低读次数。
2. **云托管服务被隔离** `ResourceIsolated`：整个 `stain` 服务 503。

> **状态（2026-07-18 充值后）**：两项均已恢复——`detail` 返回 `Status: normal`，本地端到端实测 `/api/balance` 返回 `balance:3`（首登赠送生效）、`/api/recharge/packages` 正常返回种子套餐，确认数据库配额已恢复。重新部署后做线上复核即可。

## 5. 安全设计（满足需求强制项）

1. **严禁前端改次数**：余额只存云数据库，前端 `useBalance` 只读；`useAIStream` 已移除本地 `consume()`/`hasQuota()` 记账。
2. **原子扣次**：`where({user_id, balance: gte(cost)}).update(inc(-cost))`，并发下不会扣成负数或超扣。
3. **幂等防重**：ledger `_id = ${user_id}::${request_id}`，前端每次生成 UUID `request_id`；网络重试命中既有 doc 直接返回，绝不二次扣费。
4. **余额不足不调 AI**：`consume()` 返回 `INSUFFICIENT` 时直接 402（含 `need`），不消耗任何 AI token，前端提示「还差 X 次」并弹充值。
5. **AI 失败自动退次**：流开始前抛错或流传输中抛错 → `refund()` 退回并写退款流水，标记原流水 `refunded`，防重复退。
6. **支付回调验签 + 重复只充一次**：`markOrderPaid()` 对已 `paid` 订单直接返回（幂等）；缺微信商户参数时不伪造成功。
7. **密钥走环境变量**，不写死。
